It's a Briefing Doc
CLAUDE.md is loaded at the start of every session — like a daily briefing. Claude reads it before doing anything, so it always knows your stack, rules, and conventions.
📂
Hidden by Design
The .claude folder starts with a dot — intentionally hidden from casual users. ls won't show it. Use ls -la to see it. This is where your project's AI intelligence lives.
⚡
Closer = Stronger
Project-level rules override global rules which override auth rules. The closer to your actual codebase, the more control you have. Project always wins conflicts.

Priority Hierarchy — Who Wins Conflicts?
Three levels. Closest to your project wins.
📌
Project-Level (.claude/)
Rules inside your specific project folder. Closest to Claude. Always highest priority.
WINS
🏠
Global (~/.claude/)
Preferences in your home folder. Applies across ALL projects unless overridden.
MID
🏢
Auth / Organization Level
Company-wide rules. Useful for enforcing standards across all engineers at a company.
BASE
💡 Real example: Auth says "always use Python". Your project CLAUDE.md says "this is a Java 8 project". Claude will use Java 8 — and may even ask you to clarify if they conflict.

What Goes in CLAUDE.md?
Six sections that turn Claude into a project expert
🧱 Stack & Architecture
Language, framework, DB, deployment. "React frontend, Node.js API, PostgreSQL, deployed on AWS."
📏 Coding Standards
Naming conventions, indent size, camelCase vs snake_case. Claude will follow these in every file it touches.
⚙️ Common Commands
npm run dev, docker compose up, run tests. Claude knows these without you typing them every time.
🚦 Behavior Rules
"Never commit to main." "Always write tests." "Ask before deleting files." Hard constraints.
🌿 Git Conventions
Branch naming, commit message format, PR description style. Keeps your repo history clean.
🏗️ Project Context
What is this app? Who uses it? Any important quirks? Helps Claude make smarter decisions.
📋 Loaded every session✅ No need to repeat yourself
⚖️
5 Golden Rules for CLAUDE.md
What experienced Claude Code users actually follow
1
Keep it under 200 lines. Long files get ignored. Every line costs tokens. Lean CLAUDE.md = smarter Claude.
2
Move details to Skills. Detailed workflows belong in .claude/skills/ not CLAUDE.md. Keep this file as a high-level briefing.
3
Use CLAUDE.local.md for personal rules. This file is gitignored — safe for personal API keys, personal style preferences, local machine paths.
4
List your most-used commands. Add npm run dev, docker compose up, migration commands. Claude will always know how to run your project.
5
Let Claude help write it. Ask Claude to generate a CLAUDE.md from your existing codebase. It scans your files and creates a perfect briefing doc in seconds.

Custom Commands — Your Personal Slash Commands
Add anything to .claude/commands/ and invoke it with /command-name
1
Create file
.claude/commands/code-review.md
2
Write the prompt
"Review this file for improvements..."
3
Open new session
Claude scans .claude/commands/
4
Use /code-review
Available as a slash command!
📋 /push-to-github
Stage, commit with message, push to origin. Reusable git workflow.
🔍 /code-review
Review any file for bugs, patterns, improvements. Project-aware.
🚀 /init-context
Load project context, ask clarifying questions, set coding mode.
🏠 Global commands go in ~/.claude/commands/ — available in ALL projects. Project commands go in .claude/commands/ — only this project.
