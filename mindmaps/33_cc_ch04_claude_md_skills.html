CLAUDE.md = Team Standards
One markdown file that every agent reads first. Coding conventions, banned patterns, preferred tools — set once for the whole team.
📚
Skills = User Manuals for Tasks
A skill.md file is a reusable playbook. Teach Claude Code how to do a specific task once — it follows the same steps every time.
🔢
Token Efficiency is Everything
Skills 2.0 loads only the skill header into context until needed — saving tokens on every request that doesn't use that skill.
📋
Anatomy of a CLAUDE.md File
The five sections every good project configuration file should have
🏢
Project Overview
What the project does, tech stack, primary language, team context. Claude reads this first to understand the domain before touching any code.
✅
Coding Standards & Conventions
Python type hints required, PEP 8, docstrings for all public functions, max line length 100. These are applied to every file Claude writes or edits.
🔧
Preferred Tools & Frameworks
Use Polars not Pandas, pytest not unittest, Pydantic for validation. Claude will choose these automatically without you specifying them each time.
🚫
Prohibitions / Never-Do
Never use print() for logging, never hard-code credentials, never commit to main directly. Guardrails that protect your production systems.
🔗
Skills References
Pointers to skill.md files the agent should load. @skills/migrate_db.md — Claude loads these playbooks when matching tasks arise.
💡
⭐ The Team Multiplier Effect
Two developers (Rahul and Rohan) using Claude Code without CLAUDE.md get two different coding styles. With one shared CLAUDE.md, every AI-generated line follows the same standards — code reviews become trivial.

Anatomy of a skill.md File
Four components that make a skill reusable, testable and shareable
📝 Metadata Header
Name, description, when to trigger. This is the only part always in context (Skills 2.0).
---
name: fetch_api_data
trigger: "fetch" OR "API call"
---
📖 Step-by-Step Instructions
Numbered steps Claude must follow exactly. Think of it as a recipe — not guidelines, actual steps.
1. Validate endpoint URL
2. Use requests with retry
3. Log errors to Datadog
📌 References & Sub-Skills
Pointers to other skills, helper files, or external docs this skill depends on. Builds composable skill trees.
@skills/auth_headers.md
@utils/retry_logic.py
✅ Validation / Tests
Expected outputs or assertions Claude should verify after completing the skill. Quality control baked in.
assert response.status == 200
assert len(data) > 0

Skills 2.0 — What Changed
Smarter context loading, auto-generation & evaluation
📁 Skills 1.0
Full skill body always in context
Manually written skill.md files
No quality evaluation built-in
Costs tokens even when unused
🆕 Skills 2.0
Only metadata in context; body lazy-loaded
Skill Creator plugin auto-writes skills
Built-in eval: runs skill, grades output
Token cost only when skill is triggered

Token Economy — Why Metadata Matters
Every token in context costs money and slows response. Skills 2.0 optimises this.
Full body
~2400 tokens per skill
Metadata
~120 tok
Smart load
Body on-demand only
10 skills × 2400 tokens = 24,000 wasted tokens per request if all loaded. Metadata-first saves ~90%.
💡
Metadata in context — just the skill name, trigger keywords, and one-line description; tiny token footprint
💡
Body lazy-loaded — full instructions loaded only when the trigger keyword appears in your request
💡
Compound savings — the more skills you define, the more pronounced the savings vs always-in-context approach

Skills System Pitfalls
What goes wrong when CLAUDE.md and skills are misused
!
No CLAUDE.md — two developers get two different code styles; review time doubles because nothing is consistent
!
Vague skill steps — "handle errors gracefully" is not a step; "wrap in try/except and log to Datadog with error code" is
!
Too many skills always in context — 20 full skill bodies × 2000 tokens = 40K wasted tokens per request before you even start
!
Skipping eval — a skill that runs but produces subtly wrong output is worse than no skill; always test the happy path
🚫 Vague Steps 🚫 Token Waste 💡 Always Eval
